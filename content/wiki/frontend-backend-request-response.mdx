## 1. 프론트엔드: 요청 만들기

모든 건 프론트엔드 코드 한 줄에서 시작한다.

```js
const res = await fetch("https://api.example.com/users/42", {
  method: "GET",
  headers: {
    "Authorization": `Bearer ${token}`,
    "Accept": "application/json",
  },
});
const user = await res.json();
```

## 2. 보내기 전: 브라우저 캐시 확인

요청을 네트워크로 흘리기 전에, 브라우저는 "이거 전에 받은 적 있지 않나?"를
확인한다. 캐시가 살아 있으면 네트워크를 아예 타지 않는다.

<Steps
  steps={[
    {
      title: "fresh면 캐시에서 바로",
      desc: "Cache-Control 유효기간이 안 지났으면 네트워크 없이 끝.",
      detail: (
        <p className="m-0">
          이전 응답에 <code>Cache-Control: max-age=3600</code> 이 있었다면, 1시간
          안에는 서버에 가지 않고 캐시 값을 그대로 쓴다 (200 from cache).
        </p>
      ),
    },
    {
      title: "stale하면 재검증 (ETag → 304)",
      desc: "유효기간이 지났으면 버리지 않고 '아직 그대로냐'고 묻는다.",
      detail: (
        <p className="m-0">
          이전 응답의 <code>ETag: "abc123"</code> 을 다음 요청에{" "}
          <code>If-None-Match: "abc123"</code> 으로 붙여 보낸다. 안 바뀌었으면 서버는
          본문 없이 <code>304 Not Modified</code> 만 돌려주고, 브라우저는 캐시 본문을
          재사용한다 — 다운로드 비용 0.
        </p>
      ),
    },
  ]}
/>

<Aside term="ETag">
  서버가 응답에 붙여주는 <strong>그 데이터의 버전 지문</strong>이다. 본문 내용으로
  만든 짧은 식별자(<code>"abc123"</code> 같은 값)라, 내용이 바뀌면 ETag도 바뀐다.
  브라우저는 캐시한 ETag를 다음 요청에 <code>If-None-Match</code> 로 되보내
  "내가 가진 버전 그대로냐"를 묻고, 같으면 서버는 본문 없이 <code>304</code> 만
  돌려준다. 덕분에 안 바뀐 데이터를 다시 내려받지 않는다.
</Aside>

  left="Browser"
  right="API Server"

    { from: "right", label: "304 Not Modified", note: "본문 없음 → 캐시 재사용" },
  ]}
/>

캐시에 없거나 갱신이 필요하면, 이제 진짜로 백엔드까지 가야 한다.

## 3. 네트워크 계층 (요약)

프론트와 백엔드 사이에는 물리적인 네트워크가 있다. 요청 한 번 보내려면
이 준비 과정을 거치지만, 프론트 개발자 입장에서는 대부분 자동으로 일어난다.

    이 모든 걸 처음부터 하는 게 아니다.

## 4. CORS Preflight (다른 출처일 때)

여기가 프론트↔백 통신에서 가장 자주 막히는 지점이다. 프론트(`app.example.com`)와
API(`api.example.com`)의 **출처(origin)가 다르기** 때문에, 브라우저는 본 요청
전에 "이 요청 보내도 돼?"라고 먼저 물어본다.

<Callout type="deep" title="언제 Preflight가 붙나">
  <p className="m-0">
    단순 요청(GET, 기본 헤더만)은 바로 보낸다. 하지만 <code>Authorization</code>
    같은 커스텀 헤더가 있거나, <code>PUT/DELETE/PATCH</code> 거나,
    <code>Content-Type: application/json</code> 으로 본문을 보내면 — 본 요청
    <strong>전에</strong> <code>OPTIONS</code> 사전 요청이 자동으로 먼저 나간다.
  </p>
</Callout>

    맞으면, 데이터는 서버에서 정상으로 왔어도 <strong>브라우저가 응답을 막는다</strong>.
    "서버는 200을 줬는데 프론트에서 CORS 에러"가 나는 이유다. 해결은 프론트가
    아니라 <strong>백엔드의 CORS 설정</strong>에서 한다.

## 6. 백엔드 → 프론트: HTTP 응답

처리가 끝나면 백엔드는 **상태 코드 + 헤더 + 본문**으로 응답을 돌려준다.

```http
HTTP/2 200 OK
Content-Type: application/json; charset=utf-8
Cache-Control: max-age=3600
ETag: "def456"
Access-Control-Allow-Origin: https://app.example.com

{ "id": 42, "name": "Zaman", "role": "student" }
```

## 7. 프론트엔드: 응답 처리

응답이 도착했다. 프론트엔드는 이걸 받아 화면에 반영할 준비를 한다.

```js
if (!res.ok) {
  // 4xx, 5xx 분기 처리
  if (res.status === 401) return refreshTokenAndRetry();
  throw new Error(`요청 실패: ${res.status}`);
}
const user = await res.json();   // 본문 역직렬화
setUser(user);                   // 상태 갱신
setLoading(false);               // 로딩 끄기
```
