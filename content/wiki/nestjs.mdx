# NestJS — Node 백엔드에 '구조'를 강제하는 프레임워크

Node로 백엔드를 짜면 보통 Express부터 만난다. Express는 가볍고 자유롭다 —
라우트 하나에 함수 하나 붙이면 끝이다. 작은 프로젝트엔 이 자유가 축복이다.
그런데 기능이 늘고 사람이 늘면 그 자유가 저주가 된다. **어디에 무슨 코드를 둘지
정해진 규칙이 없어서**, 누구는 라우트 파일에 DB 쿼리를 박고 누구는 따로 빼고,
폴더 구조도 사람마다 다르다. 시간이 지나면 "이 로직이 대체 어디 있지?"가
일상이 된다.

NestJS는 이 문제를 **구조를 강제**해서 푼다. "요청을 받는 곳(컨트롤러)", "비즈니스
로직을 두는 곳(서비스)", "관련된 것들을 묶는 단위(모듈)"의 자리를 정해두고,
개발자가 그 틀 안에 코드를 채우게 한다. 자유를 일부 줄이는 대신, **누가 짜도
비슷한 모양**이 나와 큰 팀·큰 코드베이스에서 길을 잃지 않게 한다. Spring을
써봤다면 익숙할 발상이다 — 실제로 NestJS는 Spring에서 많은 걸 빌려왔다.

바뀌는지 보면 답이 보인다.

자리와 경계를 미리 그어준다.

## 2. 요청이 계층을 타고 흐르는 모습

→ 데이터 계층을 차례로 지난다. 각 층이 한 가지 책임만 갖는다.

## 3. 핵심 장치: 의존성 주입(DI)

NestJS를 NestJS답게 만드는 한 가지를 꼽으면 **의존성 주입(Dependency
Injection)**이다. 이름은 거창하지만 발상은 단순하다. 컨트롤러가 자기 손으로
서비스를 만들지(`new Service()`) 않고, **필요하다고 선언만 하면 프레임워크가
만들어 넣어준다.** 이 작은 차이가 테스트와 확장을 크게 바꾼다.

  steps={[

      detail: (
        <p className="m-0">

          <code> UsersService</code> 인스턴스를 찾아 자동으로 주입한다. 만드는 책임이
          쓰는 쪽에서 <strong>프레임워크로 옮겨간다</strong>.
        </p>
      ),
    },
    {

      detail: (
        <p className="m-0">
          테스트할 땐 진짜 DB를 쓰는 서비스 대신 <strong>가짜 구현을 주입</strong>해,
          컨트롤러 로직만 떼어 검증한다. 직접 <code>new</code> 로 만들었다면 이
          교체가 불가능하다. DI의 가장 큰 실익이 여기 있다.
        </p>
      ),
    },
    {
      title: "재사용·공유가 쉽다",
      desc: "같은 인스턴스를 여러 곳이 공유(싱글톤)할 수 있다.",
      detail: (
        <p className="m-0">
          Nest는 주입할 대상을 보통 <strong>한 번 만들어 공유</strong>한다. 어디서
          요청하든 같은 인스턴스를 받으니 자원을 아끼고 상태를 일관되게 관리한다.
        </p>
      ),
    },
  ]}
/>

<Callout type="deep" title="데코레이터로 선언하는 스타일">
  <p className="m-0">
    NestJS는 <code>@Controller</code>, <code>@Injectable</code>,{" "}
    <code>@Get</code> 같은 <strong>데코레이터</strong>로 "이건 컨트롤러다", "이건
    주입 가능한 서비스다"를 표시한다. 명령형으로 일일이 등록하는 대신{" "}
    <strong>선언</strong>해 두면 프레임워크가 읽어 조립한다. Spring의 애너테이션과
    같은 발상이라, 한쪽을 알면 다른 쪽이 빨리 익는다.
  </p>
</Callout>

<Callout type="warn" title="구조의 값은 규모가 정한다">
  <p className="m-0">
    이 모든 구조는 <strong>공짜가 아니다</strong> — 작은 API 하나에도 모듈·
    컨트롤러·서비스를 갖춰야 해 초기 비용이 있다. 그래서 <strong>아주 작은
    프로젝트엔 과할</strong> 수 있고, 팀이 크고 오래 갈 코드일수록 그 구조의 값이
    커진다. "구조를 강제한다"는 장점과 "가볍지 않다"는 단점은 한 동전의 양면이다.
  </p>
</Callout>

정리하면 NestJS는 Express의 자유가 비용이 되는 지점에서, 모듈·컨트롤러·서비스로
자리를 정하고 의존성 주입으로 조립해 **일관성·테스트 용이성·낮은 결합**을
가져온다. 그 대가로 무게와 학습 비용을 치른다. 작은 일엔 과하지만, 크고 오래
가는 백엔드엔 길을 잃지 않게 하는 골격이 된다.
